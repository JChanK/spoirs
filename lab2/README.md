# Лабораторные работы №1 и №2
## TCP и UDP клиент-сервер с передачей файлов

---

## Структура проекта

```
lab1/
  server.py        — TCP сервер (ЛР №1)
  client.py        — TCP клиент (ЛР №1)

lab2/
  udp_protocol.py  — Протокол надёжной доставки UDP (скользящее окно, ACK)
  server_udp.py    — UDP сервер (ЛР №2)
  client_udp.py    — UDP клиент (ЛР №2)
```

## Лабораторная работа №1 — TCP

### Запуск

**Терминал 1 — сервер:**
```bash
python server.py
# Слушает на 0.0.0.0:9090
```

**Терминал 2 — клиент:**
```bash
python client.py                    # localhost
python client.py 192.168.1.10 9090  # удалённый хост
```

### Команды клиента

| Команда | Описание | Пример |
|---|---|---|
| `ECHO <текст>` | Сервер вернёт текст | `ECHO Hello World` |
| `TIME` | Текущее время сервера | `TIME` |
| `UPLOAD <путь>` | Загрузить файл на сервер | `UPLOAD /tmp/test.bin` |
| `DOWNLOAD <имя>` | Скачать файл с сервера | `DOWNLOAD test.bin` |
| `CLOSE` | Закрыть соединение | `CLOSE` |

### Тестирование через telnet / netcat

```bash
# telnet
telnet 127.0.0.1 9090

# netcat
nc 127.0.0.1 9090
ECHO hello
TIME
CLOSE
```

### Утилиты диагностики

```bash
# Сканирование портов (nmap)
nmap -sT -p 9090 127.0.0.1

# Список открытых сокетов (netstat)
netstat -tnp | grep 9090

# Альтернатива netstat (ss)
ss -tnp | grep 9090
```

### Тест докачки (обрыв соединения)

1. Начать `UPLOAD` большого файла (> 10 МБ)
2. Во время передачи прервать сервер (`Ctrl+C`) или обрезать сеть
3. Перезапустить сервер
4. Повторно выполнить `UPLOAD` того же файла — сервер ответит `OFFSET N`, передача продолжится с байта N

## Лабораторная работа №2 — UDP

### Отличия от ЛР №1

| Параметр | TCP (ЛР №1) | UDP (ЛР №2) |
|---|---|---|
| Протокол | TCP | UDP |
| Порт по умолчанию | 9090 | 9091 |
| Гарантии доставки | ОС | Реализованы вручную |
| Механизм надёжности | — | ACK + повтор + скользящее окно |
| Размер буфера | 4096 байт | 8192 байт (оптимальный) |
| Размер окна | — | 16 пакетов |

### Запуск

**Терминал 1 — сервер:**
```bash
cd lab2
python server_udp.py
# Слушает на 0.0.0.0:9091
```

**Терминал 2 — клиент:**
```bash
cd lab2
python client_udp.py                     # localhost
python client_udp.py 192.168.1.10 9091   # удалённый хост
```

### Команды — те же, что в ЛР №1

```
ECHO Hello
TIME
UPLOAD /path/to/file.bin
DOWNLOAD file.bin
CLOSE
```

Загруженные файлы сохраняются в `downloads_udp/`.
Файлы на сервере хранятся в `server_files_udp/`.

## Тестирование скользящего окна и устойчивости UDP

### Симуляция потерь пакетов (Linux — iptables / tc)

```bash
# Добавить 20% потерь пакетов на loopback
sudo tc qdisc add dev lo root netem loss 20%

# Проверить передачу — клиент должен справиться за счёт повторов
python client_udp.py
> UPLOAD test.bin

# Убрать эмуляцию потерь
sudo tc qdisc del dev lo root
```

### Симуляция потерь через firewall (DROP без уведомления)

```bash
# Заблокировать UDP порт 9091
sudo iptables -A INPUT -p udp --dport 9091 -j DROP

# Снять блокировку
sudo iptables -D INPUT -p udp --dport 9091 -j DROP
```

### Тест докачки UDP

1. Запустить `UPLOAD` большого файла
2. Остановить сервер или заблокировать порт во время передачи
3. Восстановить сервер / снять блокировку
4. Повторно выполнить `UPLOAD` — файл продолжит загружаться с сохранённого оффсета

## Сравнение производительности TCP vs UDP

Для сравнения скоростей использовать один и тот же файл (рекомендуется ≥ 50 МБ):

```bash
# Генерация тестового файла
dd if=/dev/urandom of=test50mb.bin bs=1M count=50

# TCP
python client.py
> UPLOAD test50mb.bin
# Запомнить скорость: X КБ/с

# UDP
python client_udp.py
> UPLOAD test50mb.bin
# Ожидаемо: скорость UDP > TCP × 1.5 на локальной сети
```

**Ожидаемый результат:** UDP быстрее TCP за счёт отсутствия алгоритма медленного старта, контроля перегрузки и меньшего накладного трафика служебных пакетов.

## Архитектура протокола UDP (udp_protocol.py)

### Формат заголовка пакета (12 байт)

```
+--------+--------+--------+--------+--------+--------+--------+--------+
|  type  | flags  |    window (2B)   |         seq (4B)                  |
+--------+--------+--------+--------+--------+--------+--------+--------+
|              length (4B)          |        payload...                  |
+--------+--------+--------+--------+...
```

| Поле | Размер | Описание |
|---|---|---|
| `type` | 1 байт | Тип пакета: DATA/ACK/CMD/CMDACK/FIN |
| `flags` | 1 байт | Флаги (резерв) |
| `window` | 2 байта | Размер окна отправителя |
| `seq` | 4 байта | Порядковый номер пакета |
| `length` | 4 байта | Длина полезной нагрузки |

### Типы пакетов

| Константа | Значение | Назначение |
|---|---|---|
| `PACKET_TYPE_DATA` | 0x01 | Данные файла |
| `PACKET_TYPE_ACK` | 0x02 | Подтверждение DATA |
| `PACKET_TYPE_CMD` | 0x03 | Командная строка |
| `PACKET_TYPE_CMDACK` | 0x04 | Подтверждение CMD |
| `PACKET_TYPE_FIN` | 0x05 | Завершение передачи |

### Алгоритм скользящего окна

```
Отправитель:
  base=0, next=0, window=16
  while base < total_chunks:
    send chunks [next .. base+window)
    wait ACK(base)  → base++
    on timeout: retransmit [base..next)

Получатель:
  on DATA(seq):
    send ACK(seq)
    buffer if seq > expected
    flush buffer while expected in buf
```

## Вопросы для защиты (ЛР №2)

1. **Оптимальный размер буфера UDP** — 8192 байт. Объяснение: MTU Ethernet = 1500 байт, но с учётом фрагментации IP максимальный UDP payload без фрагментации — ~1472 байт. Для локальной сети jumbo frames позволяют до 9000 байт. Буфер 8192 байт даёт хорошую пропускную способность без избыточной фрагментации.

2. **Механизм подтверждения** — для каждого DATA-пакета с номером `seq` получатель отвечает ACK(seq). Отправитель ждёт ACK в пределах окна.

3. **Повторная передача** — если ACK не пришёл за `TIMEOUT × MAX_RETRIES` секунд, весь диапазон [base, next) отправляется повторно (Go-Back-N).

4. **Скользящее окно** — предотвращает ожидание ACK на каждый пакет перед отправкой следующего; отправитель может иметь `WINDOW_SIZE` неподтверждённых пакетов в полёте одновременно.
